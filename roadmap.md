# Word Format Fixer - 项目发展路线图 (Roadmap)

项目已完成核心架构优化与稳定性加固。以下是建议的未来发展方向，按优先级排序。

## 1. 功能增强 (Feature Expansion) 🚀

### 1.1 批量处理 (Batch Processing)
- **描述**：目前仅支持单文件处理。用户通常需要一次性修复整个文件夹的文档。
- **实现**：
  - 前端增加“选择文件夹”或多文件拖拽。
  - 后端 Python 增加批量遍历与队列处理。
  - UI 增加总进度条和处理列表（成功/失败统计）。

### 1.2 可视化对比 (Visual Diff)
- **描述**：文档修复后，用户很难直观看到具体改了哪里。提供“修改前 vs 修改后”的直观对比能极大增强用户信任。
- **实现**：
  - 利用 Python (`pydocx` 或 `mammoth`) 将 Docx 转换为 HTML 预览。
  - 前端使用 `diff` 库高亮显示差异文本或格式变化。

### 1.3 自定义规则编辑器
- **描述**：允许高级用户不写代码就能调整规则逻辑（如“将所有红字改为黑字”）。
- **实现**：
  - 前端提供基于块（Blockly）或表单的规则构建器。
  - 后端解析规则配置并动态应用。

## 2. 工程化与发布 (Engineering & CI/CD) 📦

### 2.1 自动化构建 (Automated Build)
- **目标**：生成最终用户的安装包 (`.exe`, `.dmg`)。
- **挑战**：需要同时打包 Python 环境（使用 `PyInstaller`）和 Electron 应用（使用 `electron-builder`），并处理路径依关系。

### 2.2 自动更新机制
- **目标**：应用发布后，用户能自动检测并下载新版本。
- **技术**：集成 `electron-updater`。

## 3. 用户体验升级 (UX Improvements) 🎨

### 3.1 暗黑模式 (Dark Mode)
- **描述**：支持系统级暗黑模式切换。
- **实现**：利用 `main.css` 变量 (`var(--bg-color)`) 和 CSS Media Queries。

### 3.2 真正实时的进度反馈
- **描述**：目前的进度条是模拟的。
- **实现**：后端处理长文档时，通过 `stdout` 实时流式传输处理百分比，前端实时渲染。

## 4. 技术栈演进 (Tech Stack Evolution) 🛠️

### 4.1 迁移至 TypeScript
- **理由**：虽然已修复 Lint，但 JS 的类型安全依赖 JSDoc。迁移到 TS 能提供编译期保障。

### 4.2 引入前端框架 (Vue/React)
- **理由**：随着 UI 复杂度增加（如 Diff 视图、批量列表），手动操作 DOM (`renderer.js`) 会变得难以维护。

## 建议下一步
推荐优先进行 **1.1 批量处理** 或 **2.1 自动化构建**，这也通常是实用工具最急需的功能。
